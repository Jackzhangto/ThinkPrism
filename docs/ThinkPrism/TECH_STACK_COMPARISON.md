# 技术栈对比：React/Vite/TS vs Vanilla JS

## 1. 对比概览

| 特性 | Vanilla JS (原生 JavaScript) | React + Vite + TypeScript |
| :--- | :--- | :--- |
| **开发难度** | **低 (初期)** - 无需配置，直接写 HTML/JS | **中 (初期)** - 需配置构建工具，学习组件化 |
| **维护成本** | **高 (长期)** - 代码量增加后，状态管理混乱，UI 更新困难 | **低 (长期)** - 组件化复用，类型安全减少 Bug，状态管理清晰 |
| **性能** | **极高** - 无运行时开销，体积最小 | **高** - 现代 React 性能优化好，构建产物经过压缩 |
| **UI 开发效率** | **低** - 手动操作 DOM (innerHTML, createElement)，繁琐易错 | **极高** - 声明式 UI，所见即所得，生态库丰富 (Tailwind, Antd) |
| **代码质量** | **难以保证** - 弱类型，易产生运行时错误 | **高** - TypeScript 强类型约束，编译期发现错误 |
| **热更新 (HMR)** | **无** - 修改代码需手动刷新插件/页面 | **完美支持** - Vite 提供秒级热更新，开发体验极佳 |

## 2. 深度分析

### 2.1 Vanilla JS (原生开发)

**适用场景**：
- 极简单的插件（如：仅有一个按钮，点击后执行一段脚本）。
- 对包体积有极致要求（< 10KB）。
- 无复杂 UI 交互。

**对于 ThinkPrism 的劣势**：
- **UI 状态同步难**：ThinkPrism 的 Popup 需要展示多 Tab（场景选择）、动态列表（推荐模式）、搜索过滤。用原生 JS 手动更新 DOM 会非常繁琐且容易出错。
- **数据流混乱**：需要在 Storage、Popup、Content Script 之间传递数据，原生 JS 缺乏清晰的数据流管理，容易导致状态不一致。
- **缺乏类型提示**：Chrome API 参数复杂，没有 TS 提示，开发效率低且容易写错 API 名称。

### 2.2 React + Vite + TypeScript (推荐方案)

**适用场景**：
- **复杂 UI 交互**：ThinkPrism 需要 Tab 切换、搜索、动态列表渲染、卡片展示。
- **长期维护项目**：类型系统和组件化架构让代码易于理解和扩展。
- **多人协作/开源**：规范的代码结构方便社区贡献。

**对于 ThinkPrism 的优势**：
- **组件化架构**：可以将 `ModeCard` (模式卡片), `SearchBar` (搜索栏), `SceneSelector` (场景选择器) 封装为独立组件，复用性强。
- **声明式 UI**：只需关注数据状态（如 `currentMode`），React 自动更新 UI，无需手动操作 DOM。
- **TypeScript 支持**：Chrome Extension API (`@types/chrome`) 提供完整的类型定义，代码补全极其强大，大幅减少查文档时间。
- **Vite 构建**：
    - 支持 CRXJS 等插件，自动处理 Manifest 引用。
    - 开发时支持 HMR（热模块替换），修改 UI 样式无需重新加载插件，开发效率提升 10 倍。
- **Tailwind CSS**：配合 React 组件，可以极快地构建出现代化、美观的界面。

## 3. 结论与建议

**推荐选择：React + Vite + TypeScript**

**理由**：
ThinkPrism 不仅仅是一个简单的脚本工具，它包含了一个功能丰富的 Popup 界面（推荐引擎、模式管理、设置）。
- **为了开发效率**：React 的组件化和 Vite 的热更新将节省大量时间。
- **为了代码质量**：TypeScript 能有效避免低级错误，特别是在处理 Chrome Storage 和消息通信时。
- **为了用户体验**：React 能提供更流畅的交互体验（如即时搜索反馈）。
- **为了开源贡献**：现代化的技术栈更容易吸引开发者参与贡献。

虽然引入了构建步骤，但通过 Vite 的配置，这部分复杂度完全可控，且带来的长期收益远大于初始投入。
