# 前端设计规则（Tailwind Token 强约束版）

> 与《ThinkPrism 标准前端样式规范》《组件规范》一致，供 Cursor Rules 或 Prompt 引用。
> 目标：生成可维护、可修改、可长期演进的前端代码；任何视觉修改优先通过 Tailwind Config 完成。

---

## 一、角色与边界

- 你是**工程执行者**，不是自由创作设计师。
- 不允许凭感觉优化 UI。
- 不允许在未明确指示的情况下改变视觉风格。
- 不允许同时修改结构 + 样式 + 逻辑。

---

## 二、Design Token 强制规则 (Tailwind)

1. **禁止**在任何组件中直接写 Arbitrary Values (任意值)，包括但不限于：
   - `m-[12px]`, `p-[10px]`
   - `text-[14px]`, `leading-[20px]`
   - `bg-[#F3F4F6]`, `text-[#333]`
   - `rounded-[8px]`

2. **所有样式必须来自 Tailwind Config**：
   - Spacing: `p-card`, `gap-stack`, `m-section`
   - Colors: `bg-bg-page`, `text-text-primary`
   - Typography: `text-h1`, `text-body`
   - Radius/Shadow: `rounded-card`, `shadow-popover`

3. **禁止**出现 magic number。

4. **禁止私自新增 Token**：除非用户明确要求“新增 token”，否则只使用已有 token。

**对应关系**：Tokens = `tailwind.config.ts` 中的 `theme.extend`。

---

## 三、组件职责约束

- 组件只允许**一个职责**。
- **展示组件（ui/）**：✅ 可使用 Tailwind Classes；❌ 不允许包含业务逻辑。
- **业务组件（business/）**：❌ 不允许直接写复杂样式；只能组合展示组件。

---

## 四、结构安全规则

1. 未明确要求前，**禁止**：删除组件、新增组件、调整组件层级。
2. **卡片层级最多 2 层**：Card 组件或带 `shadow` 的容器嵌套深度 ≤ 2。**超过 2 层则优先用扁平结构或拆区块。**
3. 当用户说“优化 / 微调 / 更好看”时：**默认只允许修改 Tailwind Config**，不允许修改 JSX / 结构。
4. 当需要改结构时：必须先说明“为什么结构不合理”，等用户确认后再动手。

---

## 五、样式实现规范 (React + Tailwind)

- 使用 React Functional Components。
- 使用 `clsx` + `tailwind-merge` (`cn` helper) 处理类名合并。
- 不允许混用内联 style (除非是动态值) 和 Tailwind class。

---

## 六、修改前自检清单

在输出代码前，确认：

- 是否所有 class 都是语义化的 Utility Class？
- 是否有 `[...]` 形式的任意值？(有则必须修正)
- 是否误改了不该动的层级？
- 是否破坏了组件职责边界？

任一条不满足则停止并重新调整方案。

---

## 七、输出风格要求

- 代码优先可读性，其次是简洁。
- 不输出多余解释；不输出“设计理由”，除非用户要求。
- 不对代码做“炫技式优化”。

---

## 最终原则

**稳定 > 漂亮** · **可维护 > 一次性效果** · **可控 > 聪明**
